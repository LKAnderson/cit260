Week 2

Last week, we wrote three programs that all used a Java API (or standard library) method to print string literals (or, "hard coded" strings) to the console. This week we're learning about how to use variables to store values in our program that we can reuse. A variable is an identifier (or a name) given to a location in memory so that you can remember it more easily. Variables have a data type associated with them so that the computer can know how you want it to interpret the value stored in the memory location associated with the identifier.

There are integers (byte, short, int, long) and real numbers (float, double), true/false (boolean), single-letter characters (char) and strings (you already know about these). In addition to these "primitive" types, we have object types (such as Date). Really, Strings in Java are not technically primitive types, but are actually objects. That will be important to remember later.

Computers don't do much more than, well, compute. Variables (and the identifiers we give to them) allow us to express some very complex computations in terms that are easy to read.

Some Pro Tips

Here are a couple of tips from the trenches that might make a difference for you this week.

  • Although identifiers can start with a '$', we just don't do this in Java. It actually makes things harder to read, and it makes your code look like a Linux Bash script.
  • Give your variables descriptive names. Not too short, not too long. Single-letter names are only appropriate in short for-loops as the loop counter (eg, for (int i=0; i < 3; i++)...), and in very well-known mathematical formulas where everyone knows what the letters mean (eg, a2 + b2 = c2, the Pythagorean theorem). Other than these situations, please give your variable a name that describes what it holds. For example, a variable containing today’s date could be given the name "today" or "todaysDate" (apostrophes are not allowed in Java identifiers).
  • Although Java allows us to declare multiple variables on the same line of code (eg, int var1 = 5, var2 = 6), we've collectively decided as an industry that this makes things harder to read later, so we just don't do it. Each variable deserves it's own line.
  • Pay attention to Section 2.18 in the chapter. There are some good pitfalls there. If you happen to fall into any other pits during your work this week, please post it to the discussion board!
