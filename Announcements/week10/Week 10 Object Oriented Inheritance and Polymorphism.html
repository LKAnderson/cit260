Object Oriented Inheritance and Polymorphism

After talking about abstraction, encapsulation, association, composition and all that stuff last week, we will finish our introduction to Object Oriented Programming by learning about inheritance from one class to another, and some of the wonderful software design possibilities this opens up for us.

There is truly too much in this chapter for me summarize in one announcement. But I'll try..

When modeling real world things in classes and objects, we often find that certain related classes have common elements. We don't want to repeat ourselves, so we take those common elements and we put them in a "base" class, and then we inherit from (or "extend") the base class to define the parts that are not common between them.

For example, let's consider a data model for buildings in a city. We have the following types of buildings (keeping it simple):

  • Houses
  • Apartment Buildings
  • Shopping Centers
  • Office Buildings

Each of these types of buildings has an address, and maybe the number of tenants. So let's define a base class, called Building. Each of the other types of buildings will have other characteristics that are unique to only some of them, so we could end up with UML that looks like this:

Here's how to read this UML:

  • Inheritance (or extending) is denoted using an arrow from the child to the parent (or "base") class. The arrowhead is a hollow triangle.
  • While defining our model, we found that the commercial buildings have something else in common that a house doesn't have: a property management company (I know, I know -- rental homes might have a property manager, too. We're keeping this example simple).
  • They all have a way of calculating their occupancy. But it may be different for each type of building. 

There is an "is-a" relationship that follows the arrows in the diagram. A Mall "is-a" ManagedBuilding, which itself "is-a" Building. A House "is-a" Building. But a House is not a ManagedBuilding.

A child class inherits everything from the parent class. A Mall has everything defined in the Building and ManagedBuilding classes. And it doesn't have to redefine them. They just get inherited from their base class and are available for use in the child class.

The is-a relationship allows us to do this: 

ArrayList<Building> buildings = new ArrayList<Building>();
    buildings.add(new House());
    buildings.add(new Apartments());
    buildings.add(new Mall());
    buildings.add(new Offices());

Since they're all Buildings, we can keep them all in a list of Building objects. There's a trick to this, though. When we treat a House like a Building, we can only call the methods of the Building class on the House class. That means anything that a House has that is not common to all Buildings will not be available until we tell Java that we're dealing with a House object.

Building building = buildings.get(0);
    if (building instanceof House) {
        House house = (House)building;
        // Now we can do House-y stuff...
    }

This is typecasting, just like any other casting we have already seen. If the building is not an actual House object, though, we'll get a runtime error if we try to cast it to a House. That's why we test it beforehand using the instanceof operator.

Method Overriding

Method overriding (not quite the same as overloading) involves a child class providing its own version of a method that its parent defined. Notice the getOccupancy() method. Each type of building might have to provide its own style of counting occupancy. 

All Building objects have a getOccupancy() method. But the way occupancy is calculated might be different for each type of building. But since building has a getOccupancy() method, and all of the children of Building inherit it, we know that any kind of Building object will give us an answer back if we call that method on it. This is known as Polymorphism (or "multiple bodies").

However, check out this loop:

for (Building building: buildings) {
        System.out.println(building.getOccupancy());
    }

This loop will actually call the version of getOccupancy() from each child class. This is known as dynamic binding. It means that the program figures out which version of the method to call at runtime, based on the object it has. 

It is not magic. It's just some cool data structures and a little bit of extra code injected by the Java compiler when it compiles your program. We won't get into those data structures in this class. But just know that it's not magic. 

Remember:

Overloading = multiple methods in the same class with different inputs.

Overriding = same method in a child class; same inputs, but different implementation.
